1. 三家 Provider 折叠为 config 驱动的单一 fetch

Strong in-process
services/GlmUsageProvider.kt · KimiUsageProvider.kt · MiniMaxUsageProvider.kt · UsageParser.kt · ServiceProvider.kt(Providers) · domain/Models.kt(ServiceProviderInfo) · ui/UsageViewModel.kt(providerOptions, credentialFor) · widget/QuotaListRemoteViewsService.kt(providerLabelOf)
Before · shallow + depth 错位

KimiUsageProvider (同模板)
MiniMaxUsageProvider (同模板)
↓ 每家都调
UsageParser.parseGlm/Kimi/MiniMax (真 depth，但隐形)
⚠ 元数据发散 6 处：Models.kt / Providers.when / VM.providerOptions / Factory.providerLabelOf / credentialFor / parseXxx
After · 一处 fetch + config 表

glm/kimi/minimax: url·headers·credType·brandColor·parse

Problem

三家 Provider 重复同一 5 步 fetch 模板（~70 行）；真 depth 在 UsageParser 却隐形；加服务商要改 6 处。

Solution

Provider 变 ServiceProviderConfig 数据表（url/headers/credentialType/parse）；一个 fetch 实现；Providers.all() 集中元数据。

  • Wins
  • • 删 ~70 行重复 boilerplate
  • • locality：加服务商一处，不再发散 6 处
  • • depth 错位修正：Parser 成为显式深度杠杆
  • • 顺手删 testConnection 死方法（S9）

3. 刷新状态在 ViewModel 与 Worker 间分裂

Worth exploring in-process
ui/UsageViewModel.kt(refreshServices map) · widget/QuotaRefreshWorker.kt · services/UsageRefreshService.kt(stoppedFor / consecutiveFailures)
Before · 状态分裂（潜在 bug）
VM.refreshServices map
stoppedFor=AUTH · 节流 · 退避（clear/remove 不一致）
Worker 每 30min
new RefreshService（状态从 0 开始）
⚠ VM 停止的 AUTH 账户，Worker 继续重试 → 风控/浪费
After · 持久化"应跳过"决策
RefreshCoordinator（或 cache 元数据）
should-skip 持久化
VM + Worker
共享跳过决策
✓ AUTH 账户两处都停；rename 不丢节流
Problem

Worker 每个 cycle new 全新 RefreshService —— VM 的 stoppedFor / 节流 / 退避全丢；AUTH 失败账户被无限重试（真 bug-prone，非理论）。

Solution

最小改动：Worker 读 cache 的 errorCode，跳过 AUTH/NO_PLAN/UPSTREAM_CHANGED 账户。彻底：RefreshCoordinator 持久化 should-skip 状态，两入口共享。

  • Wins
  • • AUTH 账户停止无意义重试（修潜在 bug）
  • • locality：节流/停止决策一处
  • • rename 不再 clear 整个 map 丢节流

4. 内联单 adapter 的 CacheStorage seam

Problem

CacheStorage 只一个 adapter（假想 seam）；UsageCache object + 参数注入是混合模式；testConnection 是 dead method。

Solution

UsageCache 变 class 持 Context，内联 PrefsCacheStorage；删 CacheStorage interface（第二个 adapter 出现再抽）；删 testConnection。

  • Wins
  • • 删一个假想 seam（少一层间接）
  • • UsageCache 模式统一（不再 object+注入混用）
  • • 删 testConnection 死方法
⚠ 对比 HttpExecutor（也单 adapter）—— 那个保留，因为它给 Provider 测试真 test leverage；CacheStorage 的 test leverage 弱。
Top recommendation

先做候选 1（Provider config 化）

明天就要加 Kimi/MiniMax 真机验证 —— 此时做 config 化收益最直接：加服务商从改 6 处变成 config 表加一行，顺带删 70 行重复 boilerplate、把 UsageParser 的真 depth 显式化、删 testConnection 死方法。时机最好、风险最低、leverage 立刻兑现。

候选 2（AccountRepository）同样 Strong，但更偏内部整理（widget/worker 重复），可紧随其后。候选 3 修潜在 bug（Worker 重试 AUTH 账户），建议在多账户上线前做。候选 4 是清理项，随时可做。
