## 1. 产品概述

GLM Quota Widget 是一个本地优先的 HarmonyOS 轻量工具。用户在 App 内配置自己的 **GLM Coding Plan 专属 API Key** 后，可在 App 和桌面服务卡片中查看最近一次成功查询到的套餐用量。

产品不提供模型对话、代码生成、代理转发或账号管理；它只解决一个问题：**无需打开电脑或登录网页，也能快速确认 5 小时额度和周额度还剩多少。**

### 1.1 一句话价值主张

在 HarmonyOS 桌面上快速查看 GLM Coding Plan 的 5 小时与周用量，数据本地缓存、Key 不上传、离线仍可查看最后结果。

### 1.2 目标

| 目标 | 成功标准 |
|---|---|
| 快速可见 | 用户添加服务卡片后，3 秒内能看到最近用量或配置引导 |
| 低操作成本 | 配置完成后，查看核心额度不超过一次桌面滑动或一次点击 |
| 本地优先 | 不要求注册产品账号；无自建服务端 |
| 安全可控 | 用户可随时查看 Key 脱敏标识，并一键清除 Key 与缓存 |
| 低频稳定 | 网络波动时展示最后一次有效数据，而不是空白或不断报错 |

## 2. 用户与使用场景

### 2.1 目标用户画像

**个人开发者 / 重度 AI Coding 用户**

- 已订阅 GLM Coding Plan；
- 通过 Claude Code、Cline、Roo Code、OpenCode 等工具使用套餐；
- 经常接近 5 小时或周额度上限；
- 希望在离开电脑时也能查看剩余额度；
- 在意 API Key 不被第三方服务保存。

### 2.2 核心场景

| 场景 | 用户目标 | 产品行为 |
|---|---|---|
| 开始编码前 | 判断是否适合发起一个耗量大的任务 | 查看桌面卡片中的 5h / 周用量 |
| 外出时 | 不打开电脑也能确认套餐剩余 | 从桌面查看上次同步结果 |
| 额度接近耗尽 | 决定等待恢复、换低消耗模型或暂停任务 | 看到明确的低额度状态与重置时间 |
| 网络暂时不可用 | 知道最近一次状态而非得到空白页 | 显示缓存数据及“更新时间/可能过期” |
| Key 更换或失效 | 快速恢复可用查询 | 设置页重新配置并测试连接 |
| 不再使用 | 清除所有本地痕迹 | 设置页一键清除 Key、缓存和设置 |

### 2.3 非目标用户

- 需要团队多成员额度汇总、审计与权限管理的企业管理员；
- 希望将套餐 Key 提供给远程 SaaS 服务托管的用户；
- 需要实时（秒级）监控、复杂报表或预算预测的用户；
- 希望通过该 App 调用 GLM 模型完成编码任务的用户。

- 用户账号、手机号登录、社交登录；

- 通过普通模型请求推断套餐剩余额度；
- 高风险的接口逆向与绕过认证方案；
- 服务端推送、广告、付费订阅与数据分析 SDK。

## 4. 产品原则

1. **本地优先**：数据与 Key 默认留在用户设备。
2. **最少输入**：首次配置只要求区域与 Key。
3. **一眼看懂**：优先展示百分比、剩余状态、重置时间与更新时间。
4. **状态诚实**：数据过期、认证失败、接口变更必须明确说明，不伪装为实时。
5. **低频克制**：默认不高频后台请求，不制造额外网络或上游压力。
6. **兼容可替换**：直连功能是实验性实现；未来可切换为桥接数据源而不改变用户体验。

## 6. 页面需求

### 6.1 首次配置页

#### 目标

让用户无需理解内部实现即可安全完成配置，并在保存前验证 Key 是否可读取用量。

#### 页面结构

```text
← 配置 GLM Coding Plan

套餐区域
[ 中国站 / 国际站 ]

Coding Plan API Key
[ 输入框：粘贴或输入 Key                         ]
仅用于本机查询，不会上传至开发者服务器

[ 测试并保存 ]

提示：请使用“个人编程套餐”或“团队编程套餐”中对应的专属 Key。
```

#### 交互规则

1. 区域为必选项，默认值可根据系统区域或用户手动选择，但必须允许修改。
2. Key 输入框默认隐藏字符，支持粘贴与临时“显示/隐藏”。
3. 点击“测试并保存”后：
   - 显示加载状态，禁用重复提交；
   - 使用一次查询验证连接；
   - 成功时保存 Key 至安全存储，并将查询结果写入缓存；
   - 进入首页并提示“已更新桌面卡片”；
   - 失败时保留用户当前输入，显示本地映射错误文案。
4. 成功后不再显示完整 Key，只显示脱敏标识，例如 `****...ABCD`。

#### 输入与校验

| 项目 | 规则 |
|---|---|
| Key 为空 | 禁止提交，提示“请输入 Coding Plan API Key” |
| Key 格式明显无效 | 仅做基础长度/空白校验，不假设固定格式 |
| 认证失败 | 提示“Key 无效、已失效，或不属于可用的 Coding Plan” |
| 无套餐 | 提示“该 Key 未关联可查询的 Coding Plan 套餐” |
| 网络异常 | 提示“网络连接失败，请检查网络后重试” |
| 响应不兼容 | 提示“暂时无法解析用量，请更新 App 或稍后再试” |

### 6.2 首页 / 用量详情页

#### 目标

展示当前最重要的用量信息，提供手动刷新，且让用户明确数据的新鲜度。

#### 正常状态布局

```text
GLM Coding Plan                         [刷新]
Pro · 直连实验性

5 小时额度                         58% 剩余
████████████░░░░░░░░
预计于 今天 18:20 恢复

本周额度                           32% 剩余
██████░░░░░░░░░░░░░░
预计于 7 月 19 日 09:10 恢复

最近更新：今天 15:32
数据仅保存在本机
```

#### 展示规则

| 元素 | 规则 |
|---|---|
| 套餐名称 | 上游可获取时显示；否则显示“GLM Coding Plan” |
| 进度主文案 | 默认显示“剩余 xx%”，与进度条颜色一致 |
| 5 小时窗口 | 始终放在周窗口之前 |
| 重置时间 | 有值时显示相对时间 + 精确时间；无值时显示“重置时间暂不可用” |
| 更新时间 | 显示最近成功更新时间，不能使用失败请求时间替代 |
| 直连标记 | 文案为“直连实验性”，可点击查看兼容性说明 |
| 刷新按钮 | 请求中显示加载状态；10 秒内禁止重复触发 |

#### 用量颜色规则

| 剩余比例 | 状态 | 推荐颜色 | 文案示例 |
|---|---|---|---|
| > 50% | 充足 | 绿色 | “58% 剩余” |
| 20%–50% | 注意 | 蓝色或黄色 | “32% 剩余” |
| 1%–19% | 紧张 | 橙色 | “12% 剩余” |
| 0% | 耗尽 | 红色 | “额度已耗尽” |

颜色不能作为唯一信息载体，必须同时保留文字与百分比。

## 7. 桌面服务卡片需求

### 7.1 小卡

适合快速确认当前余量。

```text
GLM Coding Plan                         ↻

5h  58% 剩余
████████████░░░░░░░░

周   32% 剩余
██████░░░░░░░░░░░░░░

更新于 15:32
```

### 7.2 中卡

补充套餐名称、重置时间和异常说明。

```text
GLM Coding Plan · Pro                   ↻

5 小时额度          58% 剩余
████████████░░░░░░░░
今天 18:20 恢复

本周额度            32% 剩余
██████░░░░░░░░░░░░░░
7 月 19 日 09:10 恢复

最近更新：今天 15:32
```

### 7.3 卡片交互

| 交互 | 行为 |
|---|---|
| 点击卡片主体 | 打开 App 首页 |
| 点击刷新图标 | 触发一次手动刷新；系统能力不支持时改为打开首页并自动刷新 |
| 未配置状态 | 显示“点击配置”，点击打开配置页 |
| 过期状态 | 显示最后有效数据与“数据可能已过期” |
| 认证失败 | 显示“需要重新配置”，点击打开配置页 |

### 7.4 卡片刷新原则

- 卡片只读取本地缓存与由 App 推送的 Form 数据；
- 不能因为桌面频繁重绘而触发网络请求；
- App 成功刷新、清除数据、配置变更时主动更新卡片；
- 卡片数据必须和首页一致；
- 用户须手动将服务卡片添加到桌面，App 只能引导，不能静默添加。

## 8. 刷新与提醒策略

### 8.1 刷新规则

| 触发方式 | 规则 |
|---|---|
| 用户手动刷新 | 立即请求；10 秒内的连续点击忽略或复用当前请求 |
| App 回到前台 | 最近成功时间超过 15 分钟时，静默刷新一次 |
| 后台刷新 | 用户开启后低频执行，建议 60 分钟；不保证精确触发 |
| 网络错误 | 保留缓存，15 分钟后再尝试自动刷新 |
| 连续失败 3 次 | 自动刷新暂停 6 小时，用户仍可手动刷新 |
| 认证错误 / 无套餐 | 停止自动刷新，直到用户重新配置 |
| 结构不兼容 | 停止自动刷新，提示更新 App |

### 8.2 本地通知（P1）

通知须由用户明确开启，且默认关闭。

| 事件 | 条件 | 去重规则 |
|---|---|---|
| 低额度 | 5h 或周额度首次低于阈值 | 同一窗口仅一次 |
| 额度耗尽 | 剩余变为 0% | 同一窗口仅一次 |
| 额度恢复 | 上次耗尽，刷新后恢复至 > 0% | 每次真实恢复一次 |

默认阈值为 20%，可选 10% / 20% / 30%。

通知示例：

- `GLM 5 小时额度不足：剩余 18%，预计 18:20 恢复。`
- `GLM 周额度已恢复，可继续使用。`

禁止在通知中展示 Key、账号标识或原始上游错误。

## 10. 非功能需求

### 10.1 性能

| 指标 | 目标 |
|---|---|
| 首页读取本地缓存 | ≤ 500 ms（常规设备） |
| 卡片读取已缓存数据 | ≤ 3 秒显示内容 |
| 单次网络请求超时 | 15 秒 |
| 同时进行的刷新请求 | 最多 1 个 |
| 缓存数据体积 | < 100 KB |

### 10.2 稳定性

- 任何上游请求失败不得导致应用或服务卡片崩溃；
- 原始响应字段缺失时必须安全降级；
- 卡片无法更新时不得影响缓存和首页展示；
- App 冷启动必须先展示缓存或未配置页，不能等待网络完成才显示内容。

### 10.3 隐私与安全

- Key 仅存于系统安全存储；
- 普通偏好存储、数据库、日志、通知和截图预览中不得出现完整 Key；
- 不上传用量数据和设备标识；
- 不集成无必要的统计、广告和崩溃 SDK；
- 清除数据功能必须覆盖 Key、缓存、刷新状态和本地提醒标记。

### 10.4 可访问性

- 进度条必须配文字百分比；
- 颜色状态要有文字辅助；
- 可支持系统字体放大；
- 关键按钮有可读标签；
- 错误提示简明、可操作，不堆砌技术术语。

## 11. 埋点与诊断

MVP 默认 **不接入远程埋点**。如开发阶段需要本地诊断：

- 仅记录脱敏的事件类别，如 `refresh_success`、`auth_error`、`parse_error`；
- 不记录 Key、Authorization Header、请求 URL 的敏感部分、完整响应和个人标识；
- 提供开发构建专用开关，正式发布默认关闭；
- 用户导出日志前必须明确预览并确认。

- 未配置、过期、错误状态；
- 卡片点击跳转。

**验收：** 用户手动添加卡片后，无需打开 App 即可查看上次用量。

### v0.3：稳定性版

- 前台过期刷新；

- 隐私与兼容性说明。

**验收：** 网络/认证/接口异常均能安全降级，且不泄露 Key。

## 13. MVP 验收标准

### 用户体验

- [ ] 初次用户可在 2 分钟内完成配置并看到首次用量；
- [ ] 首页清晰展示 5 小时、周额度、重置时间（若数据可用）和更新时间；
- [ ] 用户可以从卡片一眼读取两个额度的核心状态；
- [ ] 离线或请求失败时，最后有效结果仍然可见，且状态明确；
- [ ] 用户能从错误提示直接进入可执行的修复操作。

### 安全与隐私

- [ ] 完整 Key 不出现在 UI、缓存、日志、通知和普通 Preferences；
- [ ] 不发送 Key 至开发者自有服务器；
- [ ] “清除 Key 与本地数据”会将卡片恢复为未配置状态；
- [ ] 产品明确告知直连功能的实验性与兼容性边界。

### 技术质量

- [ ] 首页和卡片消费同一份规范化缓存数据；
- [ ] 任何上游格式错误不会导致页面或卡片崩溃；
- [ ] 刷新不会并发重复执行；
- [ ] 认证失败或响应不兼容时自动刷新会停止；
- [ ] 完成架构文档定义的关键单元测试。

## 14. 开发交接说明

实现时以 `ARCHITECTURE.md` 为技术约束来源，本 PRD 为交互、范围和验收依据。发生冲突时遵循以下优先级：

1. 安全与隐私边界；
2. 上游兼容性风险边界；
3. 架构中的领域模型、Provider 隔离与缓存策略；
4. 本 PRD 的界面与交互细节。

如上游正式提供面向第三方应用的稳定用量 API 或 OAuth 授权，应优先新增 Provider 并逐步替换“直连实验性”实现，不应修改页面、卡片和缓存数据模型。
