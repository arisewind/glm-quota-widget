# GLM Quota Widget — 可行性评估与需求疑问清单

- 评估对象：[PRODUCT.md](PRODUCT.md)（PRD v1.0）与 [ARCHITECTURE.md](ARCHITECTURE.md)（架构 v1.0）
- 评估日期：2026-07-17
- 目的：在投入开发前，确认可行性边界、澄清需求与设计中尚未明确的问题

---

## 一、总体可行性结论

**结论：文档质量高、产品定义清晰、架构合理；MVP 在"假设上游用量查询通路可用"的前提下工程可行。但存在一个前置的根本性风险，必须在写第一行 Provider 代码之前关闭。**

### 优点（可以直接落地的基础）

1. **范围克制**：MVP 的 P0/P1/P2 划分清楚，"明确不做"清单（账号、云端、登录、逆向、推送、广告）写得很到位，避免了过度设计。
2. **架构分层正确**：Provider 隔离 + 领域模型 + 缓存 + Form 数据模型的四层划分，使未来切换数据源时 UI/卡片/缓存不需要重构。这是本项目最有价值的设计决策。
3. **风险边界意识强**：把直连定义为"实验性数据源"，要求上游字段只出现在 `DirectKeyUsageProvider` 内，字段变动时只改 Provider——这套约定能在上游漂移时把爆炸半径控制住。
4. **安全设计完整**：Key 安全存储、脱敏 `****...ABCD`、最小权限、不进日志/缓存/通知、清除数据覆盖全链路，符合"本地优先"的承诺。
5. **降级策略完善**：节流(10s)、并发锁、连续失败 3 次暂停 6 小时、认证/解析失败停止自动刷新、离线展示最后有效数据——可用性设计成熟。

### 必须先解决的前置风险（见下文 Q1）

整个 P0 的核心功能是"读取套餐用量"，但两份文档都承认：**未确认存在面向第三方移动 App、公开版本化且长期稳定的用量查询 REST API**。如果这个接口实际不存在或不可用，直连方案就不成立，MVP 的验收标准（"成功读取后可断网查看"）无法达成。这是 go/no-go 的前置条件。

## 二、需求与设计疑问清单

> 标注说明：🔴 = 阻塞型（不解决无法启动开发）；🟡 = 决策型（影响实现方向，需产品/技术拍板）；🟢 = 细节型（可实现时再定）。

### A. 数据来源与上游接口

#### 🔴 Q1. GLM Coding Plan 到底有没有"用量查询 API"？这是整个项目的命门

PRD 与架构反复用"实验性""未确认存在公开 REST API"描述直连数据源。但 PRD 同时把"高风险接口逆向与绕过认证方案"列为**明确不做**。这就形成了一个矛盾：

- 不做逆向 / 不抓包；
- 又没有官方公开的用量查询 API 文档；
- 那 `DirectKeyUsageProvider` 的请求路径、请求头、响应字段**从哪里来**？

**需要明确回答**：
1. 智谱 GLM Coding Plan 是否提供官方的用量/额度查询接口？官方文档链接？
2. 如果有，它是面向第三方 App 开放的，还是仅限官方客户端/插件内部使用？
3. 如果仅限内部，本 App 调用是否违反智谱服务条款？
4. 如果完全没有公开接口，那"直连"方案是否本质上无法合规实现？

> 在 Q1 没有答案前，建议**不要进入 Phase 1 编码**。可行的验证动作：用一个真实 Coding Plan Key，手动构造一次用量查询请求，确认能拿到结构化的 5h/周额度数据。

#### 🟡 Q2. "套餐区域"（中国站 / 国际站）对应的端点配置是什么？

SetupPage 要求选择区域，架构 4.3.1 要求 Provider "按用户选择的套餐区域获取端点配置"。但端点配置（base URL、鉴权方式、路径）在文档中没有给出，且完全依赖 Q1 的结论。

**需要补充**：中国站与国际站的用量查询 base URL、是否同一套接口、Key 是否跨站通用。

#### 🟡 Q3. "Coding Plan 专属 API Key"的获取入口与格式

- 这个 Key 和普通 GLM API Key（`open.bigmodel.cn` 控制台签发的那种）是同一个，还是 Coding Plan 专属签发？
- 用户在哪个页面/入口获取？PRD 应该给出引导链接，否则首次配置的"测试并保存"会卡在用户找不到 Key。
- PRD 写"不假设固定格式，仅做基础长度/空白校验"——但完全没有格式约束，无法在 UI 层做任何前置拦截，所有无效 Key 都会走到一次真实网络请求才发现。是否至少给一个前缀/长度提示？

#### 🟡 Q4. GLM Coding Plan 的额度模型本身是否如文档所述？

PRD 假设存在"5 小时窗口额度"和"周窗口额度"，且每次都带"预计重置时间"。需要确认：
1. 5h/周双窗口模型是否是 GLM Coding Plan 的真实规则？
2. 额度计量单位是什么（token？调用次数？还是混合）？
3. 上游响应是否真的提供精确的 `nextResetTime`？PRD 多处依赖它，但又在展示规则里写"无值时显示'重置时间暂不可用'"——暗示这个字段可能经常缺失。如果重置时间经常拿不到，首页/中卡的一大块信息就空了。

### B. HarmonyOS 平台能力

#### 🟡 Q5. 服务卡片的"点击刷新"是否真能触发 App 服务？

PRD 7.3 / 7.4 自己也写了"如系统 Form 交互能力允许""不能保证"。HarmonyOS Form Kit 对卡片内交互触发宿主应用逻辑的支持程度需要确认。如果系统不支持，小卡/中卡的 `↻` 刷新图标就只能退化成"打开 App 后自动刷新"，体验打折。这属于 P0 范围，需提前验证。

#### 🟡 Q6. 后台低频刷新在 HarmonyOS 4.2 上是否可靠？

PRD 8.1 / 架构 5.1 都很诚实地说"不保证精确触发"。但 P1 的"自动刷新"卖点是"外出时不打开电脑也能看"。需要确认：
1. HarmonyOS 4.2 上，Form 应用能注册的 BackgroundTask 类型与最小间隔；
2. 系统在 Doze/省电模式下是否会冻结后台刷新；
3. 如果后台刷新基本不可靠，是否应该把"自动刷新"默认关、对外只承诺"前台刷新 + 手动刷新"？

#### 🟡 Q7. SecureCredentialStore 用 HarmonyOS 的哪个具体能力？

架构 4.5 写"安全凭据/加密存储能力"，但没指明 API。HarmonyOS 上候选有 HUKS（统一密钥管理）、通用加密存储、Credential Manager 等，能力与版本绑定。需要选定一个并确认 4.2+ 支持，否则安全存储这一 P0 项的技术可行性悬空。

#### 🟢 Q8. 服务卡片对自定义进度条/颜色的渲染能力

PRD 定义了绿/蓝黄/橙/红的颜色规则和进度条。需要确认 HarmonyOS Form（ArkTS 卡片）能否渲染自定义彩色进度条、是否适配深色模式。如果卡片能力受限，可能要降级为纯文字百分比（PRD 已要求"颜色不能作为唯一信息载体"，所以降级是安全的）。

### C. 产品逻辑与交互

#### 🟡 Q9. testConnection 与 fetchUsage 的关系

架构 4.3 把 `testConnection()` 和 `fetchUsage()` 定义为两个方法，又在 8.1 要求"连接测试应尽可能复用一次实际查询的有效结果，避免测试一次+查询一次"。
- 这两个方法到底是各自独立请求，还是 `testConnection` 内部就是调一次 `fetchUsage`？
- 如果是后者，接口为何要拆成两个？是否应合并为"尝试查询 → 成功即连接成功"？

#### 🟢 Q10. "额度恢复"通知的去重逻辑边界

PRD 8.2：额度恢复"每次真实恢复一次"。但如果额度在阈值附近反复横跳（恢复→消耗→恢复），去重窗口是什么？"同一窗口仅一次"中"窗口"指的是 5h 窗口还是周窗口？跨窗口如何重置计数？

#### 🟢 Q11. 多账户（P2）下卡片显示哪个账户？

P2 多账户"独立 Key、缓存与卡片切换"，但桌面服务卡片同一时间只能绑一套数据。多账户时：
- 卡片默认显示哪个账户？
- 用户如何在卡片层快速切换，还是必须进 App 切换"当前账户"？
这条会显著影响多账户形态的交互设计，建议即使做到 P2 也要先想清楚。

#### 🟢 Q12. UPSTREAM_CHANGED 后的恢复路径

响应结构变化时提示"升级 App"。但 App 升级后的新解析逻辑只能通过发版带来——意味着上游一改字段，App 就得紧急发版追赶。HarmonyOS 应用市场的审核周期会让这个恢复路径偏慢。是否考虑：
- 把字段映射规则做成可配置/可热更新的轻量资产（即便不走服务端，也可以走应用内资产包）？
- 还是接受"必须发版"，并在文档里把响应时间预期写明？

#### 🟢 Q13. 缓存 schemaVersion 的迁移规则

架构 4.5 提到"使用 schemaVersion 迁移或清除缓存"，但没说什么情况下迁移、什么情况下直接清除。建议补一条决策规则（例如：字段新增→迁移；字段语义变更→清除并重新拉取）。

### D. 合规、风险与市场

#### 🔴 Q14. 用第三方 App 携带用户 Key 调用 GLM Coding Plan，是否违反智谱服务条款？

即使 Q1 证明接口存在，也要确认：智谱是否允许用户把自己的 Coding Plan Key 放进第三方 App 去查询用量？很多厂商的 ToS 禁止 Key 用于非官方客户端。如果违反：
- 上架 HarmonyOS 应用市场可能被驳回；
- 智谱有权封禁相关 Key。
这是和 Q1 同级的合规命门，建议在开发前向智谱确认或查阅其服务条款。

#### 🟡 Q15. BridgeUsageProvider 的前提是否成立？

Phase 4 / P2 的桥接模式依赖"电脑端官方用量查询插件"。但这个"官方插件"是否存在、能否被用户自建服务读取数据，同样没有验证。如果电脑端也没有官方的用量查询能力，那桥接模式作为"直连失败后的退路"也不成立，整个产品的兜底方案就空了。建议同步验证。

#### 🟡 Q16. 目标用户规模是否足以支撑产品？

同时满足"HarmonyOS 用户 + GLM Coding Plan 订阅者 + 经常触及额度上限 + 在意 Key 不外传"的交集可能很小。如果这是个人/学习项目，无所谓；如果期望公开分发并持续维护，建议先做一次粗略的用户规模估算，决定投入级别。

#### 🟢 Q17. "直连实验性"在应用市场审核中的接受度

带"实验性、可能随时不可用"标签的工具类 App，在 HarmonyOS 应用市场的审核与分类上是否有特殊要求？建议提前了解市场审核规则。

## 三、建议的下一步行动顺序

1. **关闭 Q1 + Q14（前置验证）**：用一个真实 Coding Plan Key 手工验证用量查询通路，并确认是否符合智谱条款。这是决定"做不做"的关卡。
2. **关闭 Q15（兜底验证）**：确认电脑端是否有官方用量查询插件，评估桥接退路。
3. **关闭 Q2/Q3/Q4（数据契约）**：把端点、Key 格式、额度模型字段固化下来，作为 `DirectKeyUsageProvider` 的输入契约。
4. **平台能力 spike（Q5/Q6/Q7）**：用一个最小工程验证 Form 点击刷新、后台刷新、安全存储三项 HarmonyOS 能力是否如预期。
5. 上述 1–4 全部通过后，再按架构文档的 Phase 1→4 推进，此时文档已经足够指导实现。

## 四、给开发者的两点提醒

- 本 PRD/架构已经具备很强的指导性，**真正缺的不是设计，而是上游数据源的确定性**。文档反复用"实验性"措辞，本质是在为这个不确定性预留退路；但 MVP 验收标准是硬性的（必须能读到真实用量），所以这个不确定性必须在 MVP 启动前转为"已验证"。
- 如果 Q1 验证后发现没有合规的直连通路，建议不要硬上逆向方案（PRD 已明确禁止），而是**直接以 BridgeUsageProvider 为 MVP 核心**——即产品从"手机直连查"改型为"读取用户电脑端已查询到的用量"。这会改变产品形态，但能保住合规与可持续性。
